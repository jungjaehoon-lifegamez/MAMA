# MAMA OS 1.0 Roadmap Design Spec

## Overview

MAMA OS evolves from a multi-agent dashboard into a **local AI runtime platform** with connected memory. The core value proposition: your digital life, remembered locally, connected intelligently, always yours.

This document defines the phased roadmap from v0.15 (current) to v1.0 (general release).

## Strategic Context

### What Claude/LLMs Commoditized
- Multi-agent orchestration (sub-agent dispatch)
- Skills/plugins marketplace
- Code execution sandbox
- MCP server integration

### What MAMA OS Uniquely Provides
- **Local persistent memory** with evolution tracking (supersedes/builds_on/debates)
- **Always-on daemon** (watches, learns, acts without user prompting)
- **Messenger-native connectivity** (Kakao, LINE, Slack, Telegram, Discord)
- **AI provider independence** (memory outlives any AI service)
- **Cross-source intelligence** (code decisions + team chat + email = one graph)

### Competitive Landscape

| Competitor | Model | MAMA OS Differentiation |
|-----------|-------|------------------------|
| Supermemory | Cloud API, closed source, #1 benchmarks | Local-only, open source, messenger-native, evolution tracking |
| Mem0 | Cloud API ($24M YC), OpenMemory MCP for local | Local-first from day one, app ecosystem, not just SDK |
| Pieces | Local developer tool, 9-month memory | Developer-only, no messenger/calendar/email, no app ecosystem |
| Reor | Local note app, semantic search | Note app only, no external app connection, manual input |

**MAMA OS occupies a unique position: local + always-on + app ecosystem + messenger-native.** No competitor combines all four.

### Existing MAMA Ecosystem

Three entry points already feeding mama-core:

| Entry Point | Context | Scope |
|-------------|---------|-------|
| Claude Code Plugin | Coding decisions, architecture choices | `claude-code/{project}` |
| MCP Server | Claude Desktop interactions | `mcp-server/{session}` |
| MAMA OS Gateways | Discord, Slack, Telegram messages | `gateway/{platform}` |
| Kagemusha (private) | Kakao, LINE, Slack, Chatwork, Telegram team monitoring | `kagemusha/{channel}` |

All share mama-core. The 1.0 roadmap unifies these into a coherent platform.

## Phase 1: v0.16 — Memory Engine Completion

**Goal:** mama-core becomes a complete memory platform, not just a decision store.

### 1.1 ingestConversation Hybrid Extraction

Migrate hybrid extraction from test scripts into mama-core API.

```typescript
await mama.ingestConversation({
  messages: [...],
  scopes: [{ kind: "app", id: "slack" }],
  extract: {
    enabled: true,
    mode: "hybrid",      // code regex first, LLM supplement
    codeThreshold: 2,    // LLM when code finds < 2 facts
    batchSize: 3,        // sessions per LLM call
  }
})
```

Pipeline:
1. Code regex extraction (15+ patterns, instant, free)
2. LLM supplement for sessions with < 2 code facts (batched, persistent session)
3. Entity key generation (unique per fact, no supersedes collision)
4. Scope tagging + topic classification
5. Supersedes chain detection (same entity across sessions)

### 1.2 ingestDocument API

New API for non-conversational data.

```typescript
await mama.ingestDocument({
  type: "image" | "pdf" | "spreadsheet" | "document",
  source: { app: "gmail", id: "attachment_123" },
  content: "extracted text or LLM vision description",
  metadata: { filename: "contract.pdf", size: 1234 },
  scopes: [{ kind: "app", id: "gmail" }]
})
```

mama-core stores extracted facts only, never raw files. Connectors handle extraction, mama-core handles storage + search.

### 1.3 Memory Agent — Real-Time Observation + Extraction

The memory agent watches live conversations (Claude Code sessions, messenger chats) and automatically saves important content without explicit user commands.

```
Live conversation (Claude Code / Telegram / Web Chat)
       ↓ observes
Memory Agent
       ↓ detects important content
  save-candidate-extractor (existing code regex: decisions, preferences, changes)
       ↓ supplements
  LLM hybrid extraction (same pipeline as benchmark)
       ↓
  mama-core.ingestConversation()
       ↓
  Memory graph (auto-recalled in next session)
```

This is the same hybrid pipeline proven in MemoryBench (100% accuracy), applied to real-time conversation streams. The save-candidate-extractor already exists in `packages/standalone/src/memory/save-candidate-extractor.ts` with patterns for decisions, preferences, and changes in both English and Korean.

Key: the memory agent runs in Telegram or web chat as an observer. It does NOT interrupt the coding flow. It silently watches, extracts, and stores. The user can check what was saved via the Control Tower or by asking the agent directly.

**This is MAMA OS's killer use case.** No competitor has an agent that observes your coding sessions and automatically builds a memory graph. Supermemory and Mem0 provide APIs — they don't watch and learn.

### 1.4 Scope-Based Vector Search

Upgrade from topicPrefix string matching to proper scope-based filtering.

```typescript
// Isolated: connector sees only its data
mama.search("meeting schedule", { scopes: [{ kind: "app", id: "slack" }] })

// Cross-scope: user sees everything
mama.search("client A", { scopes: "all" })
// → Slack 3 results + Telegram 2 + Gmail 1, unified ranking
```

Implementation: extend the topicPrefix pre-filter (built this session) to scope-based filtering in vectorSearch adapter.

### 1.4 100-Question Benchmark

Complete the 100-question LongMemEval run (currently in progress).

Target: >60% accuracy (baseline v68: 50%, code-only: 40%, hybrid 10Q: 100%).

### Completion Criteria
- [ ] `ingestConversation` hybrid mode works in mama-core API
- [ ] `ingestDocument` API works
- [ ] Memory agent extracts facts from live Claude Code session
- [ ] Scope-based vector search verified
- [ ] 100-question benchmark completed, accuracy >60%

## Phase 2: v0.17 — Connector Framework + Messengers

**Goal:** Apps can register with MAMA OS and feed data to mama-core.

### 2.1 platform-core Integration

Migrate from mama-suite into MAMA OS:
- `AppManifest` — app registration contract
- `ProcessSupervisor` — lifecycle management (start/stop/restart/health)
- `AppRegistry` — registered app catalog
- `AppClient` — inter-app HTTP communication

### 2.2 mama app CLI

```bash
mama app register ./slack-connector   # read manifest, register
mama app start slack                  # supervisor starts process
mama app stop slack                   # graceful shutdown
mama app list                         # status table
mama app logs slack                   # tail logs
mama app health                       # all apps health check
```

### 2.3 Connector Contract

Every connector follows this standard:

```
Required:
  /health          → { status, lastSync, messageCount }
  /manifest        → AppManifest

Behavior:
  On start:    Register scope with mama-core
  Running:     Collect data → mama.ingestConversation() or ingestDocument()
  On stop:     Graceful cleanup

Connector collects + extracts. mama-core stores + searches.
```

### 2.4 Official Connectors (Messenger-Native)

Messengers are MAMA OS's unique strength. No competitor has these.

**Slack Connector:**
- Bot Token authentication
- Channel polling → `ingestConversation`
- File attachments → `ingestDocument`
- Thread support (thread = session)

**Telegram Connector:**
- Bot Token authentication
- Chat/group polling → `ingestConversation`
- Image/file → `ingestDocument`
- Bidirectional: MAMA OS can send alerts via Telegram

### 2.5 Existing Entry Point Integration

Unify existing MAMA ecosystem entry points:

```
Claude Code Plugin → scope: "claude-code/{project}"
MCP Server         → scope: "mcp-server/{session}"
```

Both already save to mama-core. Add proper scope tagging so cross-source search works.

### 2.6 MCP Server Bridge

For external apps (Gmail, Calendar, GitHub, Notion), do NOT build custom connectors. Instead, bridge existing MCP servers:

```
[Gmail MCP Server] → [MCP Bridge] → mama.ingestConversation()
[Calendar MCP]     → [MCP Bridge] → mama.ingestDocument()
[GitHub MCP]       → [MCP Bridge] → mama.ingestConversation()
```

The bridge is a thin adapter: reads MCP server output, converts to mama-core ingest format. Leverages the 100+ existing MCP servers without reinventing connectors.

### Completion Criteria
- [ ] `mama app list` shows registered connectors
- [ ] Slack connector: channel messages stored as facts in mama-core
- [ ] Telegram connector: bidirectional (ingest + alerts)
- [ ] `mama memory search "meeting"` returns Slack facts
- [ ] Claude Code Plugin facts searchable alongside connector facts
- [ ] MCP Bridge prototype working with at least one MCP server

## Phase 4: v0.19 — Stability + Security

**Goal:** MAMA OS is trustworthy enough to run 24/7 unattended.

### 4.1 Watchdog

```
MAMA OS daemon monitoring:
  Self:        child process crash → auto-restart (5s delay, max 3 retries)
  Connectors:  /health every 60s → 3 failures → restart → alert if still failing
  Embedding:   port 3849 health → auto-restart on failure
```

### 4.2 Audit Log

```
~/.mama/audit.log (local only, never transmitted)

Logged events:
  [memory] save/search/delete operations
  [app] connector start/stop/health/error
  [api] external API calls with auth status
  [security] auth failures, blocked IPs
```

Viewable in Control Tower UI under Security tab.

### 4.3 Graceful Restart

```
mama restart:
  1. Start new process
  2. Verify mama-core DB connection
  3. Notify connectors
  4. Terminate old process
  5. New process reconnects connectors
  → Zero memory loss, minimal session interruption
```

### 4.4 Health Score

```
mama status:
  Health Score: 98/100
    mama-core DB:       ✓ (12ms)
    Embedding server:   ✓ (45ms)
    Slack connector:    ✓ (last sync 2m ago)
    Telegram connector: ✓ (last sync 30s ago)
    Disk:               ✓ (45MB / 10GB free)
    Uptime:             ✓ (4d 12h, 0 restarts)
```

Deductions: connector down (-10), DB slow (-5), disk low (-15), recent crash (-20).

### Completion Criteria
- [ ] Watchdog auto-recovers connector crash within 10s
- [ ] Audit log records all mama-core operations
- [ ] `mama status` shows health score
- [ ] 72-hour unattended uptime test passes

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Supermemory dominates cloud memory market | Reduces MAMA OS's appeal to cloud-first users | Double down on local-only positioning. Different market. |
| MCP Bridge proves fragile with third-party MCP servers | Connector expansion blocked | Build official connectors as fallback for critical apps |
| Hybrid extraction Sonnet costs at scale | 100+ questions expensive | Expand regex coverage to reduce LLM calls. Batch optimization (done). |
| Non-developer onboarding complexity (OAuth, permissions) | Adoption barrier | Invest in guided setup wizard. Test with real non-developers. |
| mama-core DB grows too large on disk | Performance degradation | Implement fact TTL, archival, and compaction policies |

## Design Decisions Log

| Decision | Rationale |
|----------|-----------|
| Local-only, no cloud option | Privacy as core value. Differentiates from Supermemory/Mem0. |
| MCP Bridge over custom connectors | 100+ MCP servers exist. Building Gmail/Calendar/GitHub from scratch is wheel reinvention. |
| Messenger connectors built in-house | No MCP servers for Kakao/LINE. Messenger-native is MAMA OS's unique moat. |
| Remove Skills/Playground/Agent Swarm | Commoditized by Claude. Focus on what Claude cannot do. |
| Milestone-based releases | Each version independently useful. No big-bang 1.0 risk. |
| Memory-first roadmap | mama-core quality determines everything downstream. Kagemusha experience confirmed this. |
