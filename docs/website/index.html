Two actors, one frame

MAMA holds the record. The agent simulates it.

One is the substrate — a local, append-only record of what your team did and said, preserved so stronger models can re-read it tomorrow. The other is the worker — the agent that reads, simulates, infers, predicts, and acts within explicit bounds.

Substrate · MAMA

A game record of your team. Re-readable by stronger models tomorrow.

Local SQLite. Append-only. Edges instead of overwrites. Every record carries time, source refs, model run, and envelope hash — so an agent today and a stronger model next year can both re-simulate the same history.

  • 15 connectors poll chats, docs, sheets, repos, and work logs
  • Raw layer: cross-source unified search, windowing, and as-of time travel
  • Graph: entities, cases, decisions linked by typed edges (supersedes · builds_on · debates · synthesizes)
  • Provenance ledger: source refs · model runs · tool traces
  • Envelopes: scope · expiry · allowed destinations · signed
Worker · Agent

Reads. Simulates. Infers. Predicts. Acts — within the envelope.

The same model the operator already uses — not a babysat function. Three modes: Reactive (your turn) · Delegated (background investigation) · Autonomous (overnight twin prep). Outputs are simulations, inferences, decisions, drafts, and cited reports — all bounded by what the envelope allows.

  • Reads cross-source raw with time and provenance intact
  • Simulates past positions: "what was visible at moment T?"
  • Infers the tacit: patterns and contradictions in the work record the team did not catch
  • Predicts accumulating project risk before it breaks
  • Writes decisions, cases, alias edges — directly, within envelope
  • Composes cited reports as one of many deliverables

The substrate is what does not change as models improve. The agent is what gets stronger as models improve. Build the substrate; let the agents catch up.

What's different

Most AI gives you a black box. MAMA can't.

Every output the agent ships — report, decision, draft, prediction — carries the same four properties by form, not by policy. The substrate refuses to let the agent hide them.

01

Cited, not asserted.

Every claim points back to a specific raw ref. Not "according to recent activity" — the actual messages, with timestamps and authors.

Evolved, not stale.

Decisions are linked by typed edges. A new decision supersedes the old one with a reason — and the old one is still readable.

Missing, not silent.

Bounded, not unleashed.

The agent can draft a customer message — but cannot send it unless the envelope allows that destination. Boundary is a frame, not a hope.

Why local

Your records stay on your machine by default.

MAMA's substrate (SQLite + 1024-dim embeddings) lives on the laptop or server you choose, not in someone else's cloud. Provider-backed agent runs (Claude or Codex CLI subprocess) still transmit selected prompts and compiled context to the chosen model API — but the persistent record stays local.

01 · STORAGE

Local SQLite

The whole substrate is one file on disk you already trust. Open it with any SQLite client. Read it forever, with or without MAMA running.

02 · EXECUTION

Official CLI subprocess

The agent runs as claude or codex under your existing authenticated CLI. No API keys to manage. No header spoofing. ToS-compliant.

03 · OWNERSHIP

You own the record

Append-only with provenance on every write. If you stop using MAMA, the substrate is still readable and yours to keep. The record you keep is the record you control.

04 · COST

No cloud bill for memory

No per-query billing for the substrate. Your existing Claude Code or Codex subscription covers the model. Embeddings run locally on a 560 MB model.

What MAMA is not. Not employee monitoring. Not productivity scoring. Not emotion or turnover prediction. The subject is the work, not the worker — by design. The substrate forbids people-scoring at the schema level.

Install

Start local. Keep the record yours.

Runs through your existing Claude Code or Codex CLI auth — no API keys to manage and no third-party cloud holding your substrate. Agent runs still send the selected prompt and compiled context to whichever model you point at (Claude or Codex).

⚠ Run inside an isolated environment. The Claude / Codex CLI MAMA spawns has the same filesystem and network access as the shell that started it. For shared or production setups, run MAMA OS inside Docker, a dedicated VM, or a non-privileged user account so its agents cannot reach files or services you don't intend to expose.

1

Authenticate a CLI

claude auth status
2

Install MAMA OS

npm install -g @jungjaehoon/mama-os
3

Start the daemon

mama init && mama start
